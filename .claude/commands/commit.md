# /commit — Зафиксировать изменения с обновлением документации

Выполни следующие шаги **по порядку**:

## Шаг 1 — Что изменилось?

Запусти `git diff` и `git status`, чтобы понять:
- Какие файлы изменены / добавлены / удалены
- К какому ID задачи из BACKLOG.md относятся изменения (если применимо)

## Шаг 2 — Обновить документацию (только если есть что фиксировать)

**Важно:** если изменения чисто технические (рефактор, стиль, исправление опечатки) — документацию не трогай. Обновляй только тогда, когда появилось что-то новое, что кто-то другой должен знать.

### ARCHITECTURE.md — приоритет №1

Читай **раздел «Текущее состояние»** и синхронизируй его с реальным репо:

- Появились новые файлы или директории → добавь в дерево структуры
- Удалены или переименованы файлы → убери/переименуй
- Принято новое архитектурное решение (новая зависимость, смена подхода, изменение API) → добавь строку в таблицу «Зафиксированные технические решения» с датой
- Обновляй дату «Последнее обновление» при каждом изменении раздела

**Раздел «Целевой дизайн» не трогай** — он меняется только при смене архитектурного решения, вручную.

### BACKLOG.md

- Задача начата → статус `[ ]` → `[~]`
- Задача завершена → `[~]` → `[x]`, переместить в «Выполненные задачи»
- Принято ключевое решение → добавить строку в таблицу ADR

### CLAUDE.md

Проверь: появились ли в этом коммите новые соглашения, ограничения или факты о проекте, которые должны быть известны с первого открытия репо? Например:
- Зафиксирован стек (технологии)
- Изменилась структура команды или зоны ответственности
- Появилось новое критическое ограничение

Если да — обнови соответствующий раздел CLAUDE.md. Если нет — не трогай.

### Остальные docs/

| Что изменилось | Что обновить |
|---|---|
| Новый/изменённый API-контракт | `docs/REQUIREMENTS.md` — только если меняется контракт с пользователем или между компонентами |
| Изменение продуктовой логики | `docs/PRODUCT.md` — только существенные отклонения от MVP |

**Принципы:**
- Кратко — одна строка лучше абзаца
- Не пересказывай код — только решения, контракты, структуру
- Не добавляй ничего в документы, чего нет в реальности прямо сейчас

## Шаг 3 — Сформировать коммит

Сообщение в формате: `[ID] Краткое описание`

Примеры:
- `[C-02] Add intent parser for transfer/balance/topup`
- `[C-01] Fix gitignore for Android`
- `[docs] Restructure ARCHITECTURE.md into fact/design sections`
- `[chore] Add .gitignore`

Если изменений в документации не было — всё равно пиши честное сообщение о том, что сделано.

## Шаг 4 — Подтвердить

Покажи одну строку: что закоммичено и что (если что-то) обновлено в документации.
