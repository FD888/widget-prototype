# /commit — Зафиксировать изменения с обновлением документации

Выполни следующие шаги **по порядку**:

## Шаг 1 — Что изменилось?

Запусти `git diff` и `git status`, чтобы понять:
- Какие файлы изменены / добавлены
- К какому ID задачи из BACKLOG.md относятся изменения (если применимо)

## Шаг 2 — Обновить документацию (если нужно)

Прочитай текущее содержимое затронутых документов и обнови **только то, что действительно изменилось**. Не дублируй, не добавляй лишнего.

| Что изменилось в коде | Что нужно обновить |
|---|---|
| Задача завершена / начата | `BACKLOG.md` — статус задачи (`[ ]` → `[~]` → `[x]`), дата в ADR если принято решение |
| Новый endpoint / API | `docs/REQUIREMENTS.md` — только если меняется контракт |
| Новый компонент / слой / зависимость | `docs/ARCHITECTURE.md` — кратко, без пересказа кода |
| Изменение продуктовой логики | `docs/PRODUCT.md` — только существенные отклонения от MVP |

**Принципы обновления:**
- Кратко и ёмко — одна строка лучше абзаца
- Не пересказывай код — только решения и контракты
- Если изменение чисто техническое (рефактор, стиль) — документацию не трогай

## Шаг 3 — Сформировать коммит

Сообщение коммита в формате: `[ID] Краткое описание`
Примеры: `[C-02] Add intent parser for transfer/balance/topup`, `[C-01] Fix gitignore for Android`

Если изменения не относятся к задаче из беклога: `[docs] Update architecture diagram` / `[chore] Add gitignore`

Затем выполни коммит через стандартный процесс (git add → git commit).

## Шаг 4 — Подтвердить

Покажи краткую сводку: что закоммичено, что обновлено в документации.
